# ForecastIQ — Retail Sales Time Series Forecasting Dashboard
**Project by Poorani M** | Data Analyst Portfolio

---

## What This Project Does

End-to-end time series forecasting pipeline on 5 years of daily retail sales data (3 stores, 2018–2023).

**Analytical coverage:**
- Exploratory Data Analysis (seasonality, trend, promotions, COVID impact)
- Additive seasonal decomposition (trend + seasonal + residual)
- 3 forecasting models: Holt-Winters, Gradient Boosting, Seasonal Naïve
- Walk-forward (expanding window) validation — no data leakage
- 30-day ahead point forecast with 95% prediction intervals
- Model drift monitoring (rolling MAE)
- Business insights with actionable recommendations

**Tech stack:** Python · Flask · Pandas · NumPy · Scikit-learn · Plotly.js · No statsmodels/Prophet required

---

## Quick Start (3 steps)

```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Precompute analysis (runs once, ~3 min, caches to JSON)
python3 precompute.py

# 3. Start the server
python3 app.py
```

Open **http://localhost:5000** in your browser.

## Models Implemented

| Model | Type | Key features |
|-------|------|-------------|
| **Holt-Winters** | Statistical | Level α=0.3, Trend β=0.1, Seasonal γ=0.2, period=7 |
| **Gradient Boosting** | ML | 7 lag features, cyclical encoding, rolling stats, promo/holiday flags |
| **Seasonal Naïve** | Baseline | Same day last week — industry standard baseline |

All evaluated via **walk-forward (expanding window) validation** — not random split.

---

## Why Walk-Forward, Not Random Split?

Random split shuffles the date index — the model trains on **future data** to predict **past data**. This is called temporal data leakage and produces falsely optimistic metrics that fail in production.

Walk-forward validation enforces strict temporal order: training data is always before test data. The metrics you see in this dashboard reflect real deployment performance.
